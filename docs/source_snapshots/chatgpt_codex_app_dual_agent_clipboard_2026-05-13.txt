1. 目前最值得关注的开源项目
1. OpenAI Symphony：最 Codex-native 的官方架构规范
Symphony 是现在最重要的参考。它的 spec 明确说自己是一个 long-running automation service：持续读取 issue tracker，为每个 issue 创建 isolated workspace，并在 workspace 内运行 coding agent session。它要解决的核心问题正好是你现在遇到的四个：把 issue 执行变成 repeatable daemon workflow、隔离 agent execution、把 workflow policy 放进 repo 的 WORKFLOW.md、提供足够 observability 来管理多个并发 agent run。

更关键的是，Symphony spec 的 goals 已经把你前面想写的 scheduler/registry 大部分定义好了：固定 cadence polling、bounded concurrency、single authoritative orchestrator state、deterministic per-issue workspace、当 issue 状态变化后停止 active runs、transient failure exponential backoff、repo-owned WORKFLOW.md、structured logs 等。

它不是“直接装上就完全替代你现有 controller”的东西。OpenAI repo 里说可以按 spec 自己实现，也提供了 experimental Elixir reference implementation；同时明确这是 low-key engineering preview。
 所以我的建议是：

不要把 Symphony 当成产品直接无脑上生产；
要把 Symphony 当成你的 managed Codex controller 规范。
你现在自己设计的 registry/scheduler/worktree/worker result/evidence index，和 Symphony spec 的方向非常一致。区别在于：你不需要再自己发明整套术语和生命周期，直接按 Symphony 的 issue/workflow/workspace/run/session/retry model 对齐。

2. Composio Agent Orchestrator：最接近“开箱即用 coding-agent fleet manager”
ComposioHQ 的 agent-orchestrator 是目前我看到最像你想要的“多 coding agent 管理平台”的开源项目之一。它的 README 说它可以 spawn parallel AI coding agents，每个 agent 有自己的 git worktree；agent 会自动修 CI failure、处理 review comments、open PR，你从一个 dashboard 监督。它还明确支持 Claude Code、Codex、Aider，runtime 支持 tmux、ConPTY/process、Docker，tracker 支持 GitHub、Linear。

它的工作流也非常贴近你的需求：ao start 启动 dashboard 和 orchestrator agent；orchestrator 给每个 issue spawn worker，每个 worker 在 isolated git worktree 里工作；agent 读代码、写测试、创建 PR；CI failure 和 review comments 会 route 回 agent；人只在需要判断时介入。

这可能是你最该先试的项目，因为它覆盖了你不想自己写的很多东西：

dashboard
worktree lifecycle
agent session management
PR creation
CI failure reaction
review comment routing
GitHub / Linear integration
multi-agent runtime abstraction
但它未必完全满足你现在最特殊的诉求：复用现有 Codex App thread、通过 App Server 精确 turn/start、turn/steer、turn/interrupt、thread/archive 管理旧 thread。它支持 Codex，但从公开 README 看，更像是 agent/runtime 插件化运行 Codex，而不是专门复刻你当前 app-server thread registry 的控制语义。这个点需要做 spike 验证。

我的判断：

目标是“少写代码 + 有 dashboard + issue/PR/CI 工作流”
→ 先试 Composio Agent Orchestrator。

目标是“保留现有 Codex App Server 旧 thread 精确控制”
→ 可能需要给 AO 写/改一个 Codex App Server adapter。
3. Paperclip：更像“agent 公司 OS”，适合总控/治理，不一定适合底层 coding harness
Paperclip 也很值得看，尤其你已经提到 Paperclip/managed-agents 线。它的定位不是单纯 coding harness，而是“open-source orchestration for zero-human companies”：Node.js server + React UI，用来编排一队 AI agents，分配 goals，跟踪 work/costs，用 org chart、budgets、governance、goal alignment、agent coordination 来管理工作。它支持 OpenClaw、Claude Code、Codex、Cursor、Bash、HTTP 等运行时。

Paperclip 的强项在于：org chart、heartbeat、cost control、multi-company isolation、ticket system、governance。它的 README 还明确提到可以 pause/terminate agent，并且用 tickets、threaded conversations、persistent sessions、budget throttling、heartbeat 来解决“20 个 Claude Code tabs 不知道谁在干什么”的问题。

它更适合当：

人类/Lead 的上层控制台
目标/预算/治理/heartbeat 系统
多 agent 组织结构
但它不是专门为“Codex App Server thread lifecycle + worktree + PR/CI”设计的底层 coding scheduler。Paperclip 自己也说它不是 agent framework，不教你怎么 build agents，而是管理由 agent 组成的组织。

所以它对你的定位应该是：

Paperclip = 上层 mission control / governance / heartbeat / budget
AO 或 Symphony-style runner = coding execution layer
Codex App Server adapter = 具体控制 Codex threads/turns
Paperclip 也有 companies 仓库，提供 ready-to-deploy agent companies，比如 fullstack forge、security、game studio、engineering company 等，其中一些模板包含 dispatching-parallel-agents、subagent-driven-development、using-git-worktrees、verification-before-completion 等技能。
 这对你定义 lead/worker/reviewer/test_mapper 很有参考价值。

4. ORCH：最轻量、terminal-first、零云、文件状态
ORCH 是另一个很适合你的候选，尤其你现在已经有 systemd watcher、本地 controller、JSONL state。ORCH 的 README 说它可以协调 Claude、Codex、Cursor 和任何 CLI tool 并行工作，“one npm install, zero infrastructure”；安装命令是 npm install -g @oxgeneral/orch，然后在项目里运行 orch 打开 TUI。

它的 CLI 能力很像你想做的任务注册表：agent add/list/disable/enable、task add/list/assign/cancel、team create/join、goal add/list/status、msg send/broadcast、shared context、run/watch、serve、status、logs、TUI。

它还支持 headless daemon：orch serve 可以作为无 TUI daemon 跑，输出 structured JSON logs，能用 pm2/systemd/process manager 部署；它声明没有 cloud component，所有状态在 .orchestry/ 的 YAML/JSON/JSONL 里。

这个项目的意义是：如果你不想要 Postgres/Web UI/复杂 dashboard，而是想把现有 controller 替换成更成熟的 terminal-first 状态机，ORCH 可能是最低摩擦选择。

推荐场景：

你想继续本地跑；
不想引入复杂服务；
可以接受文件状态；
需要 team / task / goal / message / daemon；
不强依赖复用 Codex App 已有 thread。
潜在问题：

它可能不会天然支持你当前 Codex App Server 的旧 thread resume / steer / archive 语义；
可能需要把你的 controller 接成 ORCH 的 Codex adapter 或 shell adapter。
5. Stoneforge：角色模型很像你想要的 Director / Worker / Steward，但风险偏高
Stoneforge 的角色划分很接近你说的 Lead/Workers/Manager：Director 负责战略规划和拆任务，Ephemeral Worker 自动由 dispatch daemon 派发，在 isolated worktree 里执行、commit、push、完成或 handoff；Persistent Worker 用于手动探索；Steward 负责 merge review、文档扫描、恢复 stuck tasks、可重复维护流程。它支持 Claude Code、OpenCode、OpenAI Codex 作为 provider。

但是它自己也写得很直白：这是 early-stage experimental software；典型 session 会消耗大量 token；agent 会 autonomously run，甚至 permissions bypassed，没有 human approval gates，会读写执行并 push code。

所以 Stoneforge 很适合拿来研究架构和角色模型，但不建议你直接拿它管重要 repo，除非你把它放进非常严格的 sandbox / throwaway branch / 测试仓库里。

我的评价：

架构启发价值：高
直接替代你 controller：中等
安全默认值：偏激进
生产可控性：需要强审查
6. Vibe Kanban：好用的本地 UI / diff review surface，但不适合当核心长期控制面
Vibe Kanban 支持用 kanban issues 规划工作，为 coding agents 创建 workspaces，每个 workspace 有 branch、terminal、dev server，还能 review diffs、inline comments、内置浏览器预览、创建 PR 和 merge；它支持 Claude Code、Codex、Gemini CLI、GitHub Copilot、Amp、Cursor、OpenCode 等 10+ coding agents。

但一个关键风险是：它的 README 已经写明 “Vibe Kanban is sunsetting”，项目会继续开源和社区维护。
 所以它不适合作为你的长期核心 scheduler，但非常适合作为：

本地 visual board
worktree / branch / terminal 管理
diff review UI
browser preview UI
人工监督界面
也就是说：

Vibe Kanban 可以替代你未来想写的 dashboard；
不要用它替代 registry scheduler。
7. codex-mcp：最接近“把 Codex App Server 能力暴露成工具”的小项目
codex-mcp 是 TypeScript MCP server，包装 OpenAI Codex CLI，能 spawn codex app-server child processes，或者 fallback 到 codex exec --json，并把能力暴露成 MCP tools。它的工具快照里有 codex start new session、codex_reply continue session、codex_session list/get/cancel/interrupt/fork/clean、codex_check poll events/respond approvals 等。

它还把 Codex CLI 升级视作 protocol upgrade，并要求每次 codex --version 变化时重新跑：

codex app-server generate-json-schema --experimental --out codex-schema
然后检查 schema drift。
 这点和我们前面说“不要手写猜 API 参数，要 schema-driven”完全一致。

这个项目不一定是完整 scheduler，但它可能帮你少写最烦的一层：

Codex App Server subprocess 管理
session list/get/cancel/interrupt/fork/clean
event polling
approval/user input bridge
schema drift runbook
MCP tool surface
推荐用法：

不把 codex-mcp 当总控；
把它当 Codex App Server adapter 的参考或直接复用层。
8. cmux：不是 orchestrator，但 worktree/session 生命周期做得很干净
cmux 是 “tmux for Claude Code”，核心是每个 agent 一个 git worktree，避免多个 agent 在同一 working directory 里互相踩文件；它强调 git worktrees 共享同一个 .git database，但给每个 agent 独立目录树，避免 clone/sync，分支保持同步。

它不解决 registry/scheduler/interrupt/archive/PR review，但它可以作为你 worktree lifecycle 的极简参考。对你来说，它更像一个“不要自己写 worktree 脚本”的借鉴对象。

结论标准：

第二优先级：OpenAI Symphony
原因：这是 Codex orchestration 官方规范，方向最权威。它的 spec 已经把你要写的 scheduler/registry/retry/workspace/workflow 说清楚了。

用法不是“直接替换”，而是：

把你现在 controller 的概念重命名/对齐到 Symphony：

lane/task        → issue/workflow
thread/turn      → live session/run attempt
repo prompt      → WORKFLOW.md
state JSON       → orchestrator authoritative state
worktree         → deterministic per-issue workspace
auto-continue    → retry/continuation policy
logs             → structured observability
这样你后续即使切到别的开源项目，语义也不会乱。

第三优先级：ORCH
原因：你现在的 systemd watcher + JSONL 状态，和 ORCH 的 terminal-first/headless daemon/file-state 很像。它的 orch serve 可以无 TUI 跑 daemon，用 JSON logs，状态存在 .orchestry/，没有 cloud。

适合场景：

你暂时不想上复杂 dashboard；
你想保留本地-first；
你想很快获得 agent/task/team/goal/msg/status/logs；
你能接受状态文件而不是 SQLite/Postgres。
它的优势是迁移摩擦小。缺点是可能没 AO 那么成熟地处理 PR/CI/review routing。

第四优先级：Paperclip
原因：Paperclip 是更高层的“agent company”控制面，适合你想要的 Lead/CEO/CTO/governance/heartbeat/budget。它支持 Codex，而且解决“20 个 terminal 追踪不了”的问题。

但它不是纯 coding harness。更适合做：

上层目标管理
agent org chart
budget / cost guardrail
heartbeat
audit log
pause / terminate / governance
如果你已经在 Paperclip/managed-agents 线上推进，我会把它定位为：

Paperclip 管“谁该做什么、预算多少、什么时候 heartbeat”
AO/Symphony-style runner 管“怎么在 repo/worktree/PR/CI 里执行”
Codex App Server adapter 管“怎么精确驱动 Codex”
暂不作为核心：Vibe Kanban / Stoneforge / cmux
Vibe Kanban 很适合当 UI，但它已经 sunsetting，不建议压核心长期状态。

Stoneforge 的 Director/Worker/Steward 角色很适合借鉴，但它明确写了 early-stage、token 消耗高、默认无 approval gates、agent 会读写执行和 push code，不适合作为你重要仓库的默认控制面。

5. 最佳社区实践已经很清楚了
把这些项目放在一起看，2026 年社区共识已经不是“多开几个 agent”，而是下面这套：

1. issue / task / goal 是任务真相源；
2. 每个任务一个 isolated workspace / worktree；
3. 每个 agent 有独立 branch / PR / terminal / logs；
4. scheduler 控制 bounded concurrency；
5. agent 完成后必须给 proof of work；
6. CI failure 和 review comments 自动 route 回 agent；
7. 人只在 judgment / approval / merge 时介入；
8. workflow policy 放在 repo 内，而不是藏在聊天里；
9. 状态、日志、retry、stuck detection 必须由 orchestrator 掌握；
10. subagent 只是局部执行能力，不是长期总控。
Symphony spec 明确写了 single authoritative orchestrator state、bounded concurrency、deterministic workspaces、stop ineligible active runs、retry/backoff、repo-owned WORKFLOW.md、observability。
 AO、Vibe Kanban、Stoneforge、ORCH 也都围绕 worktree、branch、PR、dashboard/TUI、parallel agents 这些模式展开。

1. Codex subagent 的真实边界
Codex 官方现在有三个内置 agent：default、worker、explorer。default 是通用 fallback，worker 偏实现和修复，explorer 偏只读代码探索。Codex 也支持在 ~/.codex/agents/ 或项目内 .codex/agents/ 放自定义 TOML agent；每个自定义 agent 至少要有 name、description、developer_instructions，也可以覆盖 model、model_reasoning_effort、sandbox_mode、mcp_servers、skills.config 等配置。

Codex subagent 的生命周期是“父 thread 里临时 spawn 的子 agent”。它只有在你明确要求时才会 spawn；多个 agent 跑完后，Codex 会等待结果并汇总；你可以在 CLI 里用 /agent 切换查看，也可以直接让 Codex steer、stop、close 正在跑或已完成的 subagent。subagent 会继承当前 sandbox policy，父 turn 的 runtime overrides 也会重新应用到 child。

这意味着：Codex subagent 适合“当前 worker thread 内的短任务”，不适合当长期 managed agent。 它没有 Claude Code Agent Teams 那种 shared task list、mailbox、teammate 之间直接通信、团队清理机制。Codex 的全局 start/resume/interrupt/archive 还是要通过 App Server 的 thread/turn API 来做。App Server 支持 thread/start、thread/resume、thread/fork、thread/read、thread/list、turn/start、turn/steer、turn/interrupt、review/start 等控制动作。

2. 默认 subagent 够不够？
临时手动用，够。做 managed system，不够。

默认 default / worker / explorer 的问题是：

1. 角色太宽，容易漂移；
2. 没有你的 WORKER_RESULT / SUBAGENT_RESULT 输出契约；
3. 没有明确 cost tier；
4. 没有每个角色独立 sandbox；
5. 没有“只读 / 可写 / 可联网 / 可测试”的硬边界；
6. 不能表达你想要的“启动时定义角色 + 完成后回收”的协议；
7. 不方便做跨 Codex / Claude / Kimi / MiniMax 的统一调度。
所以我建议：保留默认 agent 作 fallback，但实际项目里使用 mc_ 前缀的自定义 agent。

解释：

max_threads = 4

max_depth = 1
  必须保持 1。只允许 root worker 直接 spawn subagent；

5. Codex subagent 的启动与回收协议
你说的“像 Claude Code 一样启动时定义角色，然后及时回收”，在 Codex 里要靠 父 worker prompt + custom agent + controller 状态机 配合实现。

5.1 父 worker 启动 subagent 的标准 prompt
你是 managed-codex 的当前 worker，不是全局 lead。

本轮目标：
{{task_goal}}

请启动以下 Codex subagents，并等待所有结果：

1. mc_repo_mapper
   任务：只读定位相关文件、调用链、入口、数据流。
   禁止：修改文件、运行 destructive command。

2. mc_test_mapper
   任务：找最小验证命令、现有测试覆盖、缺失测试。
   禁止：修改文件。

3. mc_risk_reviewer
   任务：从 correctness/security/regression/data risk 角度审查计划。
   禁止：style-only 评论。

要求：
- 每个 subagent 必须返回 SUBAGENT_RESULT。
- 等待所有 subagent 完成。
- 对完成的 subagent，关闭其 agent thread。
- 如果某个 subagent 卡住或跑偏，请停止它并返回 partial。
- 最终汇总成 WORKER_RESULT。
- 不要在本轮直接实现，除非我明确给出 implementation mode。
Codex 官方文档里也推荐这种模式：明确要求“每个点 spawn 一个 agent，等待全部结果，再汇总”；同时文档说明可以让 Codex steer、stop 或 close subagent threads。

5.2 实现阶段 prompt
本轮是 implementation mode。

你可以启动：
- mc_small_implementer：执行最小代码改动；
- mc_debugger：运行 focused tests / 复现命令；
- mc_risk_reviewer：实现后只读 review。

流程：
1. 先让 mc_small_implementer 做最小改动；
2. 再让 mc_debugger 跑最小验证；
3. 再让 mc_risk_reviewer 审查 diff；
4. 全部完成后关闭 subagent threads；
5. 汇总 WORKER_RESULT。

禁止：
- 不要启动超过 3 个 subagent；
- 不要让 subagent 再 spawn subagent；
- 不要改超出计划的文件；
- 不要部署、push、merge、publish。
5.3 回收标准
Codex subagent 的回收标准要写进父 prompt，也要写进 controller：

subagent 完成：
  parent 汇总 SUBAGENT_RESULT
  parent 请求 close completed agent threads

subagent 跑偏：
  parent steer: “停止当前方向，返回 partial”
  如果仍跑偏，controller interrupt parent turn

subagent 超时：
  parent 要求 stop
  controller 超过 max_turn_runtime_seconds 后 turn/interrupt

worker 完成：
  controller 读取 WORKER_RESULT
  有 evidence 才进入 next_action
  无 evidence / 无 schema / 无测试说明 → needs_retry 或 waiting_user

thread 完成：
  controller archive，而不是靠 parent 自己忘记上下文
App Server 这边要做硬回收：turn/interrupt 可以取消 in-flight turn，成功后 turn 结束为 interrupted；thread/archive 可以把 thread log 移到 archived sessions，默认后续 thread/list 不再显示。

7. Claude Code 怎么放进来
你想要的“启动时定义角色、运行中调度、完成后回收”，Claude Code Agent Teams 目前比 Codex subagent 更像这个形态。Claude Code Agent Teams 是一个 team lead 管多个 Claude Code instances，teammates 有独立 context window，可以直接互相通信，有 shared task list，可以单独和 teammate 对话；这不同于 subagent 只把结果回报给主 agent。

Claude Code Agent Teams 还支持指定 teammate 和模型、plan approval、直接和 teammate 说话、shared task list、让 teammate shutdown、cleanup team。它也有质量门 hooks，例如 TeammateIdle、TaskCreated、TaskCompleted。

但是它现在仍是 experimental，而且有几个硬限制：session resumption 不完整、task status 可能滞后、shutdown 可能慢、一个 lead 同时只能管理一个 team、没有 nested teams、teammates 不能再 spawn team、lead 不能转移。

所以我建议：

Claude Code Agent Teams
  用于：复杂任务、跨前后端、需要 teammate 互相讨论、需要 tmux 观察的工作。

Codex subagents
  用于：单个 Codex worker 内部的短期并行探索/review/test mapping。

外层 controller
  用于：长期状态、跨仓库、跨工具、启动/停止/归档/预算控制。
Claude Code 的 custom subagent 文件使用 YAML frontmatter + Markdown prompt；可以用 tools: Agent(worker, researcher), Read, Bash 这种方式限制 coordinator 只能 spawn 指定 subagent 类型，如果不列 Agent 就不能 spawn subagent；也可以在 settings 里 deny 某些 agent。

Claude Code 的权限还要特别小心：subagent 可能分别请求 permission，权限提示会变多；官方建议用 PreToolUse hooks 或 permission rules 处理，但也提醒 hook 触发 subagent 时要避免递归 loop。

你说的 “K” 我先不猜是 Kilo、Kiro、Qwen/K 还是别的 CLI，下面统一抽象成 k_cli。核心是：外层 scheduler 根据任务类型、风险、成本、上下文要求选择 runtime。

Kimi Code 官方页面说明它是 Kimi membership 的 coding-focused 能力，能放进 terminal/IDE/Kimi CLI 工作流，当前展示的模型是 kimi-for-coding powered by kimi-k2.6，能力包括软件开发、代码库分析、运行命令/处理文件/自动化 workflow、web research，并且页面示例里也提到可以 spawn subagents。

MiniMax 这边，我更建议你通过 Claude Code / Cursor / OpenCode / Kilo / 其官方推荐的工具路径 来用，而不是直接塞进最新 Codex。MiniMax M2.7 文档明确把 Claude Code 作为推荐路径，并给出 Anthropic-compatible base URL 配置；同一文档对 Codex CLI 的路径标注为 “Not Recommended”，说明它不是最佳 agent workflow，并提示最新 Codex CLI 兼容性可能有问题。

如果你要自己构建 MiniMax agent，MiniMax 官方还有 Mini-Agent demo，包含完整 agent execution loop、persistent memory、上下文管理、Claude Skills、MCP、详细日志等。这个更适合拿来做一个独立 minimax_agent runtime，而不是作为 Codex 的 subagent。

Qwen/Alibaba Coding Plan 也适合作为低成本 coding runtime。阿里云 Model Studio Coding Plan 文档列了支持的 coding tools，包括 Codex、OpenClaw、OpenCode、Claude Code、Cline、Cursor、Qwen Code、Kilo Code / Kilo CLI；它提供 OpenAI-compatible 和 Anthropic-compatible base URL，但也明确警告 Coding Plan API key 只能用于 coding tools，不能用于自动化脚本、自定义后端或非交互 batch API，否则可能被视为滥用。

所以这里要分清楚：

可以：
  让 scheduler 启动一个受支持的 coding tool CLI；
  让 CLI 在它自己的正常交互/agent 模式下使用 coding plan；
  记录 session、日志、结果。

谨慎/不要：
  把 coding plan key 直接接进你自己的后台 API 批处理；
  在无人值守 daemon 里大规模消耗 plan quota；
  让一个 agent 通过 Bash 偷偷启动另一个 agent 而不经过 registry。
Qwen Code 自身也支持在 settings.json 的 modelProviders 中配置多个 provider，并通过 /model 切换；它支持 OpenAI-compatible、Anthropic、Gemini、本地 OpenAI-compatible 等 provider，也支持 Alibaba Cloud Coding Plan 的自动配置。

12. Codex custom provider 要不要接 Kimi/MiniMax/Qwen？
我的建议是：Codex 主路径不要直接接 MiniMax/Kimi，除非你验证当前 Codex 版本和目标 provider 的 wire API 完全兼容。

Codex 本身支持 custom model providers，可以配置 base_url、env_key、headers、auth command、wire_api 等；也可以用 profiles 切换模型/provider。

但是 MiniMax 官方对 Codex CLI 路径当前标了 “Not Recommended”，并说最新 Codex CLI 有兼容问题；它更推荐在 Claude Code 或 Cursor 这类工具里使用 MiniMax-M2.7。

所以实际策略：

Codex App Server:
  主要跑 OpenAI Codex 原生模型。
  这是高可信、高控制、高 schema 的路径。

MiniMax:
  优先跑在 Claude Code-compatible / Cursor / OpenCode / Kilo / 官方推荐工具里。

Qwen / Alibaba Coding Plan:
  优先跑 Qwen Code / Kilo / OpenCode / Claude Code / Codex 支持路径。

  优先用 Kimi Code CLI 自己作为 runtime，不要塞进 Codex subagent。
13. 你真正需要的“角色启动时定义”应该长这样
外层 controller 派任务时，不是简单说“继续”，而是构造一个完整 worker launch spec。

{
  "task_id": "chatgptrest-017",
  "lane_id": "chatgptrest_main",
  "repo_id": "chatgptrest",
  "runtime": "codex_high",
  "mode": "planning",
  "role_pack": [
    "mc_repo_mapper",
    "mc_test_mapper",
    "mc_risk_reviewer"
  ],
  "allowed_actions": [
    "read_files",
    "run_safe_tests"
  ],
  "forbidden_actions": [
    "write_code",
    "git_push",
    "deploy",
    "modify_secrets"
  ],
  "timeout_seconds": 1800,
  "cleanup_policy": {
    "close_completed_subagents": true,
    "interrupt_on_timeout": true,
    "archive_on_done": false
  },
  "output_schema": "WORKER_RESULT"
}
Codex prompt 由这个 spec 生成：

你是 managed-codex worker。

本轮 launch spec：
- task_id: chatgptrest-017
- runtime: codex_high
- mode: planning
- role_pack: mc_repo_mapper, mc_test_mapper, mc_risk_reviewer
- write_code: false
- timeout: 1800s
- output: WORKER_RESULT

请启动 role_pack 中的 subagents。
等待全部完成后，关闭已完成 agent threads。
最终返回 WORKER_RESULT。
Claude Code prompt 由同一个 spec 生成：

Create an agent team with three teammates:
1. repo mapper
2. test mapper
3. risk reviewer

Require plan approval before any implementation.
Use shared task list.
When each teammate is done, ask them to shut down.
After all teammates shut down, clean up the team.
Return WORKER_RESULT.
Kimi/MiniMax/Qwen prompt 由同一个 spec 生成：

你是低风险执行 worker。
只做 repo mapping / test mapping / log summary。
禁止写代码，除非 launch spec 明确允许。
最终输出 WORKER_RESULT JSON。
这样你就实现了：角色在启动时定义；权限在启动时定义；回收策略在启动时定义；结果格式在启动时定义。

14. 收放自如的 lifecycle
统一状态机：

created
  ↓
starting
  ↓
running
  ↓
collecting_result
  ↓
closing_children
  ↓
completed
  ↓
archived
异常路径：

running
  → timeout
  → interrupting
  → interrupted
  → needs_retry / failed

running
  → user_disabled_lane
  → draining / interrupting
  → archived

running
  → output_invalid
  → needs_retry / waiting_user

running
  → unsafe_action_detected
  → interrupting
  → failed
不同 runtime 的回收动作：

cleanup:
  codex_appserver:
    soft_stop: turn/steer "stop and return partial"
    hard_stop: turn/interrupt
    archive: thread/archive

  codex_subagent:
    soft_stop: parent asks subagent to stop
    close: parent asks Codex to close completed agent threads
    hard_stop: interrupt parent turn

  claude_team:
    soft_stop: ask teammate to shut down
    cleanup: ask lead to clean up team
    hard_stop: kill process group / tmux pane
    caveat: shutdown may wait for current request/tool call

  kimi_code:
    soft_stop: send interrupt / stop instruction if supported
    hard_stop: terminate process group
    cleanup: delete temporary worktree if no diff retained

  minimax_worker:
    soft_stop: same as hosting tool
    hard_stop: terminate process group
    cleanup: delete temporary worktree / close session

  shell_runner:
    soft_stop: SIGTERM
    hard_stop: SIGKILL
Claude Code 官方也提醒：teammate shutdown 是通过 lead 发 shutdown request，teammate 可以 graceful exit；cleanup 会移除 shared resources，但如果还有 active teammates，cleanup 会失败，所以要先 shutdown teammate。

19. 最关键的原则
Codex subagent = 当前 Codex worker 内部的短期角色。
Claude Agent Team = 当前复杂任务内的协作团队。
Kimi/MiniMax/Qwen/K = 外层 scheduler 可选择的 worker runtime。
Registry = 唯一长期事实源。
Controller = 唯一有权 start/stop/archive 的总控。
Worktree = 隔离执行层。
WORKER_RESULT = 跨所有工具的统一回执。
所以：Codex subagent 要配，但不要把它配成总控。
你要的“收放自如”，不是靠某一个工具内部的 subagent 实现，而是靠：

role_pack
+ runtime_catalog
+ routing_policy
+ timeout/interrupt/cleanup
+ output_schema
+ high-model review
+ registry state machine
这样你就可以做到：

简单任务 → Kimi / MiniMax / Qwen / K
复杂任务 → Codex high / Claude Team
批量任务 → CSV subagents / cheap runtime
最终判断 → Codex / Claude reviewer
状态管理 → 你的 controller
这才是既省钱、又能控、又能真正利用你手上所有 coding plan 的方式。

10. 开源/半开源 orchestrator 为什么还是不够
我重新看了 2026 年开源 agent orchestrator 对比，结论基本一致：大多数工具已经收敛到 git worktrees，但 coordination depth 差很多。Augment 的一篇对比把工具分成三层：human-in-the-loop per-edit approval、milestone gates、spec-driven verification；它也指出多数 OSS 工具还是并行 worktree/session，真正跨服务对齐、共享 evolving plan 的能力很少。

这篇对比里对 Composio AO 的描述不错：它有 isolated worktrees、每个 agent 自己的 PR、dashboard、CI failure retry、review comment 处理。
 但我仍然不把它放在你的第一梯队，因为它更像开源/创业项目的 PR automation orchestrator，不如 GitHub/Devin/Factory/Kiro 这几个产品级平台可靠。

这篇对比也把 Claude Squad、Nimbalyst、Vibe Kanban 等归到 human-in-the-loop session management；它明确说大多数工具是 agents side by side，没有共享计划，跨 services 和 evolving specs 会变成手工协调瓶颈。

11. 成熟度和验证度怎么判断
我这次把“是否被真实使用验证”权重调高了。一个 2026 年的 AIDev 研究数据集统计了 GitHub 上 932,791 个 agent-authored PR，来自 OpenAI Codex、Devin、GitHub Copilot、Cursor、Claude Code，覆盖 116,211 个 repositories 和 72,189 个 developers。

这说明：真正有大规模公开 footprint 的还是 GitHub/Copilot、Codex、Claude Code、Cursor、Devin 这一圈。

另一篇研究分析了这些 agent PR 触发的 61,837 个 GitHub Actions workflow runs，发现不同 agent 的 CI 可靠性有明显差异，Copilot 和 Codex 的 workflow success rate 约 93% 和 94%，同时也提醒 agentic PR 频率越高，repo 层面 CI success rate 可能越受影响，需要更强 safeguards。

这对你的决策很重要：

成熟生态优先级：
GitHub / Copilot / Codex / Claude / Devin / Cursor
  > Factory / Kiro / Jules / Builder / Augment
  > 本地 OSS session managers
  > 小型自研 orchestrator

这是 最像你描述的 managed agents 产品形态 的方案。

路线 C：你想要企业级 orchestration + CLI/CI/CD
选：

Factory Droid Missions
做法：

1. 用 /missions 规划大任务；
2. 让 Mission Control 管 features / milestones / skills；
3. 用 Factory CLI 做批量迁移、CI 修复、code review；
4. 用 hooks/skills/custom droids 固化角色。
适合你如果：

你要长任务；
你要 CI/CD；
你要 structured logs / JSON events / session persistence；
你想把 agent 放进工程流水线。
这是 最像 agent-native SDLC 平台 的方案。

路线 D：你想要多 repo 一次任务
选：

Kiro Web Autonomous Agent
做法：

1. 选择多个 repositories；
2. 让 Kiro 处理一个跨 repo task；
3. 它在每个 repo 开 PR；
4. 通过 GitHub issue comments 继续给反馈。
适合你如果：

你跨 repo 工作很多；
你想用 issue label /kiro 触发；
你能接受 preview 和 vendor 约束。
路线 E：你想保留 Codex + Claude Code，并且要 Lead/spec 模式
选：

Augment Intent
做法：

1. 让 Coordinator 先产 living spec；
2. 你 review/edit spec；
3. 再让 Implementors 并行执行；
4. Verifier 根据 spec 验收；
5. Claude Code / Codex / OpenCode 作为 BYOA worker。
适合你如果：

你在 macOS；
你想要 spec-driven；
你想少写 Lead Orchestrator；
你主要复用 Codex/Claude，不强求 Kimi/MiniMax。
路线 F：你强烈想复用 Kimi、MiniMax、K、Codex、Claude
选：

CCManager 或 Nimbalyst
但要接受：

这不是完整 managed agents；
这是本地 multi-session/worktree manager。
用法：

1. CCManager 管 Claude/Codex/Kimi/Gemini/OpenCode；
2. 低风险任务派 Kimi/MiniMax/K；
3. 高风险任务派 Claude/Codex；
4. 你人工 review/merge；
5. 不要期待它自动 lead、自动验证、自动回收所有任务。
13. 我的最终推荐组合
如果是你现在这个情况，我不会再建议你继续全量自研。我的推荐是：

主控层：GitHub Agent HQ
复杂 managed agents：Devin 或 Factory
本地 plan 复用层：CCManager / Nimbalyst
高风险 review：Codex / Claude
低风险机械任务：Kimi / MiniMax / K
更具体：

1. GitHub Agent HQ 做默认 registry/control plane
   - issues
   - PRs
   - checks
   - agent sessions
   - Claude/Codex/Copilot/custom agents

2. Devin 或 Factory 处理真正复杂的大任务
   - 需要 child agents
   - 需要 coordinator
   - 需要 terminate/sleep/parallel managed sessions
   - 需要多天/多 repo 工作

3. CCManager/Nimbalyst 只处理本地 plan 复用
   - Kimi
   - MiniMax
   - K
   - OpenCode
   - Gemini CLI
   - Codex CLI
   - Claude Code

4. 你自己的 codex_managed_controller.py 停止扩张
   - 只保留旧 thread 迁移
   - 或保留特殊 Codex App Server 自动续派
   - 不再尝试做完整平台
14. 如果只选一个，我会怎么选
按你的优先级来：

最少自研、最稳生态：
→ GitHub Agent HQ / Copilot Cloud Agent

最像 managed agents：
→ Devin Managed Devins

最强工程 orchestration：
→ Factory Droid Missions

最适合跨 repo issue-to-PR：
→ Kiro Web Autonomous Agent

最适合 Codex + Claude BYOA + Lead/spec：
→ Augment Intent

最适合复用 Kimi/MiniMax/K/Codex/Claude 各种 CLI：
→ CCManager，但它不是完整 managed agents
我的真实建议是：

先试 GitHub Agent HQ。如果你觉得它的 lead/child 能力不够，再试 Devin Managed Devins 或 Factory Missions。只有当你坚持复用 Kimi/MiniMax/K 这些 plan 时，再回到 CCManager/Nimbalyst 这类本地工作台。

这条路线能最大限度避免你继续写大量 controller 代码。你现在真正需要自己做的，最多只剩：

1. 仓库/任务命名规范；
2. GitHub label/state 规范；
3. AGENTS.md / prompt / custom agent profile；
4. 哪些任务派给 Copilot、Claude、Codex、Devin、Factory、Kimi；
5. 禁派规则，例如 game lane disabled；
6. 高风险人工审批。
也就是说，不是没有替代品。只是“开箱即用的 managed agents”基本都已经是商业/官方平台了，不是开源小项目。
